# a1b44436 — Your Code Agent Can Grow Alongside You with Structured Memory (MemCoder)

## Verbatim abstract
"While 'Intent-oriented programming' (or 'Vibe Coding') redefines software engineering, existing code agents remain tethered to static code snapshots. ... To bridge this static-dynamic mismatch, we propose MemCoder, a framework designed to enable continual human-AI co-evolution. MemCoder first structures historical human experience to distill latent intent-to-code mappings from past commits. It then employs a self-refinement mechanism driven by verification feedback to correct agent behavior in real-time. Crucially, an experience self-internalization mechanism is introduced to crystallize human-validated solutions into long-term knowledge, thereby supporting sustained evolution. Experimental results on SWE-bench Verified demonstrate that MemCoder not only achieves State-of-the-Art (SOTA) performance but also delivers a 9.4% improvement in resolved rate over the general foundation model DeepSeek-V3.2."

## Verbatim contributions paragraph (intro)
"• We identify the 'interaction disconnect' in repository-level agents and propose MemCoder, a framework designed for Human-AI Co-Evolution that enables the agent to grow with the project.
• We introduce mechanisms for Human Experience Internalization: structuring historical human commits to recall past wisdom, and crystallizing human-verified solutions to accumulate current knowledge.
• We achieve SOTA performance on SWE-bench Verified, demonstrating that coupling general models with human-derived evolutionary context significantly empowers them to master complex software engineering tasks."

## Verbatim conclusion
"We proposed MemCoder to address the limitation where static code agents fail to capture the critical information embedded in the temporal evolution of projects. ... MemCoder achieves State-of-the-Art performance and effectively unlocks the potential of general models like DeepSeek-V3.2 in complex engineering tasks."

## Methodology setup (problem definition only — pre-stop)
Section 3 introduces an "Evolv" function: Π' = Evolv(Π, F(M, τ, ξ)) where F = {f_struct, f_refine, f_intern} is a composite adaptive mechanism. Three phases: (1) constructing structured memory from historical repositories, (2) retrieving context-aware experience during execution, (3) internalizing human-validated solutions for future iterations. Memory entries are sextuples m_i = (o_i, c_i, k_i, p_i, r_i, s_i) — original commit message, code changes, keywords, problem description, root-cause, solution summary. The base substrate is RAG over git commit history, layered with an iterative self-refinement loop and a write-back mechanism that adds verified solutions back into the memory store.

## 4b. Closest-prior-work identification

- **Neighbor 1: A-Mem (Xu et al., 2025, "A-Mem: Agentic Memory for LLM Agents")** — paper explicitly cites A-Mem in §2.3. A-Mem already proposes a flat, dynamically-organized agentic memory that grows by integrating new insights, with retrieval at inference and write-back of distilled experiences. MemCoder's structuring + retrieve + write-back loop is a direct application of A-Mem's pattern to git commits as the seed-memory source. The delta: MemCoder narrows the source domain (git commits with a fixed sextuple schema) and instantiates the loop on SWE-bench, but the architecture is A-Mem's.

- **Neighbor 2: SWE-Search / SWE-Gym / SWE-Synth-style commit-mining baselines (a family of 2024-2025 SWE-bench agents that retrieve from prior issues/commits to ground patch generation)** — many concurrent SWE-bench agents already mine GitHub issue/commit history as in-context exemplars (e.g., AutoCodeRover, Moatless, SWE-Search). The delta: MemCoder formalizes the schema (sextuple) and adds the cross-encoder rerank step.

- **Neighbor 3: Self-Refine (Madaan et al., 2023) + Agentless / OpenHands verification loops** — paper cites Self-Refine. The "Refining Sub-agent" that generates tests + checklist and re-prompts is the standard Self-Refine pattern with a sub-agent wrapper; verification-test generation is also done in OpenHands' default flow. Delta: introduce a separate sub-agent role rather than a single-agent self-edit loop.

- **Neighbor 4: MemoryBank / MemGPT / MemInsight (cited)** — establish the write-back-into-LTM pattern. MemCoder's "self-internalization" (M_{N+1} = M_N ∪ {m_{N+1}}) is a near-trivial set union; the conceptual operation is identical to these prior LTM update mechanisms.

- **Neighbor 5: ERM (Yan et al., 2025) and OPRO (Yang et al., 2024) — cited as "evolutionary feedback-driven" methods** — paper positions itself as integrating these mechanisms; the actual integration is engineering-level (use them together, in a SWE pipeline) rather than a new theoretical or algorithmic primitive.

Verbatim experimental result (Fig. 3): MemCoder + GPT-5.2 = 78.8%, tied with OpenHands + GPT-5.2 = 78.8% (which is the base framework MemCoder is built on). The abstract's "9.4% improvement over DeepSeek-V3.2" is real but is framed as if the framework rather than the underlying GPT-5.2 step-up drives the leaderboard claim.
- **Wrong reference class: YES.** Paper compares to "long-term memory mechanisms for LLMs" (MemoryBank, A-Mem, MemInsight) but its actual contribution is in the narrower class of "agentic SWE-bench frameworks with commit retrieval" — the paper does not benchmark against the closest concurrent methods in that narrower class on equal terms.
- **Missing the natural baseline: YES.** No ablation of "just commit-retrieval with raw commits + the same backbone, no MemCoder structuring" outside the bundled w/o ER/DSR rows; no comparison to a published commit-retrieval SWE-bench agent at the same backbone.
- **Inflated framing words: YES.** "Crucially, an experience self-internalization mechanism is introduced to crystallize human-validated solutions into long-term knowledge" — the actual mechanism is set-union (Eq. 10: M_{N+1} = M_N ∪ {m_{N+1}}).
