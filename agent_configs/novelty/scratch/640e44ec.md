## 4b. Closest-prior-work identification

- **Neighbor 1:** Creator benchmark (Qian et al., 2023) — "Creator benchmarks tool creation and execution from abstract requirements; most direct predecessor. Delta: Creator evaluates Python function creation (not MCP-standardized interfaces), uses less granular evaluation metrics, and lacks the multi-level diagnostic cascade (compliance→exec→schema→unit test→downstream)."
- **Neighbor 2:** TM-BENCH / ToolMaker (Zhang et al., 2025b) — Paper mentions this explicitly in related work footnote: "Introduces TM-BENCH and the ToolMaker agent." The exact delta is unclear without reading TM-BENCH, but paper claims distinction on: (1) requirement-driven (no specs), (2) MCP standardization, (3) diagnostic cascade.
- **Neighbor 3:** ToolLibGen (Huang et al., 2025) — "Automated tool library generation with LLMs." The delta: ToolLibGen focuses on building libraries/toolboxes from existing knowledge; Tool-Genesis evaluates creation from abstract task requirements for a specific scenario's task distribution.

## 4c. Pseudo-novelty pattern check

- **Rebrand?** No — the multi-level diagnostic cascade and requirement-driven setting are specific contributions.
- **Domain transfer?** Partially — evaluation methodology from code benchmarks (SWE-bench, APPS, HumanEval) is adapted for tool creation. The MCP context adds new evaluation challenges (schema compliance, server executability).
- **Composition without insight?** Somewhat — the benchmark combines existing evaluation components (unit tests, schema checking, execution validation). The insight is the cascaded diagnostic protocol and the oracle-normalized metric.
- **Resolution bump?** No — the main contribution is the diagnostic framework + findings, not larger scale.

## 4d. Novel framing vs. novel science

"Diagnostic benchmark" framing separates tool quality from tool utilization strategy. This enables a new kind of paper: one can now attribute LLM failures to specific pipeline stages. Whether this framing generates new scientific predictions beyond this paper is less clear.

## 4g. Positioning issues

- **Resolution bump risk:** the benchmark contribution is genuine, but the headline finding ("SOTA struggles") is expected. The paper's novelty lies in the diagnostic framing + oracle metric, not the finding itself.
- **Missing baseline:** TM-BENCH is mentioned but no direct numerical comparison is provided in the main paper (results are on Tool-Genesis itself). Without a head-to-head comparison on the same setting, it's hard to assess how different Tool-Genesis is from TM-BENCH.
- **"We propose Tool-Genesis, a diagnostic benchmark"** — the diagnostic aspect is real, but the requirement-driven aspect (no specs) is underemphasized in the abstract relative to its importance.

## 4h. Originality score rationale

The paper is a benchmark contribution in a crowded space (Creator, TM-BENCH, ToolLibGen, ToolHop all address aspects of tool creation). The genuine novelty is: (1) MCP standardization as the common interface (enabling execution-grounded validation), (2) the cascaded multi-level diagnostic protocol that separates failure causes (this is the strongest novel axis), and (3) oracle-normalized utility gap metric. These differentiators are real but incremental over the closely related Creator (2023) and TM-BENCH (2025). The finding that error cascades amplify is scientifically interesting but predictable.
