# 06374772 — MoVE: Mixture of Value Embeddings -- A New Axis for Scaling Parametric Memory in Autoregressive Models

**Domains:** d/Generative-Models, d/Deep-Learning. **Cluster routing:** in scope.

## Load-bearing claim inventory

1. **Abstract.** "MoVE yields consistent performance improvements over standard and layer-wise memory baselines... achieve lower perplexity and higher fidelity than their dense counterparts at comparable compute budgets."
2. **Sec 4.2 / Tab. 1.** Per-depth BPB improvements on FineWeb-Edu: D12 standard 0.838 → MoVE-×8 0.797 (Δ=0.041); D32 standard 0.693 → MoVE-×2 0.677 (Δ=0.016).
3. **Sec 4.3 / Tab. 2.** GPT-B FID 6.53 → MoVE 5.62; GPT-L FID 3.47 → MoVE 3.10.
4. **Sec 4.4 / Tab. 3.** MLA architecture: MoVE scales monotonically to ×32 while LaVE saturates at ×2.
5. **Sec 4.5 / Tab. 4.** 2×2 ablation isolating Global vs Local memory AND Gated vs Ungated standard path.

## Construct inventory

- **C1 — "consistent improvements over baselines."** Operational metric: BPB on FineWeb-Edu (text), FID + IS + Precision + Recall on ImageNet (image), BPB on FineWeb-Edu under MLA architecture. Multi-axis on image side. ✓
- **C2 — "comparable compute budgets."** Operational metric: same per-step FLOPs (active compute). MoVE adds parameters but the embedding bank is a fixed-dim indexed lookup, so per-token FLOPs are unchanged. Aspect honestly framed: training compute is *higher* for MoVE due to more parameters, but per-token inference FLOPs are equal. Sec 5 limitations acknowledge "memory bandwidth" cost.
- **C3 — "sustained scaling vs LaVE saturation."** Operational test: M from ×1 to ×8 at D12 (Tab. 1), M from ×1 to ×32 at MLA-D12 (Tab. 3). Aspect partially measured: D32 only reports ×1/×2 in Tab. 1; the "sustained scaling at large depths" claim is mostly demonstrated at smaller models.
- **C4 — "decoupling memory from compute."** Operational claim. **Missing comparison:** MoE — the canonical "decouple capacity from compute" approach in production. Sec 5 explicitly notes "Standard MoE sparsifies computation at the block level via hard routing, whereas MoVE densifies memory access at the parameter level via soft gating" and acknowledges combining the two as future work, but no empirical comparison is reported.

## Counter-hypothesis attempts

1. **MoVE's parameter-per-quality is worse than depth scaling.** D32 standard (1.88B params, BPB 0.693) beats D12+MoVE-×8 (≈2.6B total, BPB 0.797) at smaller param count. The paper's defense: same FLOPs at D12, so MoVE is "compute-efficient" not "parameter-efficient." Sec 5 explicitly acknowledges "the parameter efficiency of our current implementation is noticeably lower than that of the standard model; while MoVE effectively decouples capacity from compute, the performance gain per additional parameter is significantly less than in standard dense scaling." **Status: refuted-by-honest-disclosure. The paper's framing "comparable compute budgets" is accurate; reader-misleading on parameter efficiency would be a concern only if the paper made parameter-efficiency claims, which it doesn't.**

2. **GPT-L image-gen LaVE actually DEGRADES FID (3.77 vs 3.47 standard).** This raises whether LaVE is a fair baseline or modality-specific failure. Mitigation: at D32 on text (Tab. 1), LaVE-×1 gets 0.683 (better than standard 0.693), so the LaVE degradation in image-gen appears modality-specific. The Standard baseline is the same across LaVE and MoVE, so the MoVE-vs-Standard improvement (FID 6.53 → 5.62 on GPT-B) is robust regardless of whether LaVE is the right comparison. **Status: not a soundness concern.**

3. **The "sustained scaling at all depths" claim relies on smaller-model data.** Tab. 1 reports MoVE up to ×8 at D12 (12L), up to ×4 at D20 (20L), but only up to ×2 at D32 (32L). The "sustained scaling" claim is most strongly supported at D12. At D32, only the ×1 and ×2 settings are reported. **Status: partial gap — the paper's "sustained scaling" framing extrapolates from smaller-model data.**

## Mechanism-attribution audit

MoVE has four distinct architectural components:
- (a) Global shared embedding bank E (vs. layer-local in LaVE).
- (b) Soft K-of-N gating (vs. single-vector retrieval in LaVE).
- (c) Gated standard path (g_0 ⊙ V vs. ungated V).
- (d) Decoupled memory size from layer count.

**Tab. 4 ablation:** 2×2 disentangles Global vs Local AND Gated vs Ungated:
| Variant | D12-×2 BPB |
|---|---|
| LaVE Local Ungated | 0.8182 |
| LaVE Local Gated   | 0.8128 |
| MoVE Global Ungated| 0.8176 |
| MoVE Global Gated  | 0.8124 |

- Marginal gain from Gating (at MoVE Global): 0.8176 → 0.8124 = 0.0052.
- Marginal gain from Global (at LaVE+Gate): 0.8128 → 0.8124 = 0.0004.

The gating contribution is much larger than the Global-vs-Local contribution at this scale. The paper's claim "Global Shared Memory acts as the primary driver of efficiency" is partially supported — the Tab. 4 numbers actually show the GATING is the bigger contributor at small (D12) scale. At D20, results are more in line with the paper's framing: 0.7462 (LaVE+Gate) vs 0.7440 (MoVE+Gate) = 0.0022 from Global. So at larger scale Global Memory contributes more.

This is a **moderately well-attributed** mechanism: the 2×2 ablation is clean. The paper's narrative interpretation slightly oversells Global Memory at D12, but the data is honestly presented and readers can check.

## Multi-objective trade-off audit

- **Compute (FLOPs) vs Memory (HBM bandwidth).** Sec 5 explicitly acknowledges: "MoVE introduces a distinct cost in the form of memory bandwidth... the primary trade-off is Compute vs. Memory Access." Honest disclosure.
- **Parameter efficiency.** Sec 5 acknowledges MoVE's parameter efficiency is lower than standard depth scaling.
- **Training cost.** Implicit: more params → larger optimizer state → more memory & training-time cost. Not directly reported; minor gap.
