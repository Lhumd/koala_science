## Construct inventory
**Claim: "latency by up to 38%"**
- Operational metric: MoE FFN kernel latency (profiled via NVIDIA Nsight Systems)
- Abstract says "latency" without qualification
- Sec 5.3: "DES-Vote reduces MoE layer latency by up to 38.0%...Total end-to-end GPU kernel time improves by 8.2–14.3%"
- **Unmeasured aspect**: end-to-end wall-clock latency. The headline 38% applies to MoE kernel only; end-to-end improvement is 8.2-14.3%, a 3-5x gap.

**Claim: "retaining 99% of vanilla accuracy"**
- Operational metric: Relative accuracy (R.Acc) vs vanilla baseline on HumanEval, MBPP, GSM8K, MATH500
- Table 1: DES-Vote (β=0.15) on LLaDA2.0-Mini: Avg R.Acc = 99.5% ✓
- Absolute accuracy for vanilla baseline is not reported — relative accuracy reported exclusively
- **Unmeasured aspect**: absolute accuracy. Cannot assess whether "99% of vanilla" is meaningful without baseline absolutes.

**Claim: "decoupling memory overhead from degree of parallelism"**
- Operational metric: Figure 7b — average activated experts vs. block size (8/16/32/64) for DES-Vote
- Demonstrated: DES-Vote maintains roughly constant active expert count across block sizes ✓
- This specific claim is well-supported.

**Claim: multi-GPU and CPU-offloading motivation**
- Sec 1: "typical of multi-GPU and CPU-offloading environments" as the core motivation for DES
- Evaluation: "All experiments were conducted on NVIDIA B200 GPUs" — single GPU only
- **Aspect not measured**: multi-GPU and CPU-offloading scenarios are never evaluated

## Counter-hypothesis list
1. **B200 artifact**: B200 GPUs have dramatically higher memory bandwidth than H100/A100. In older-generation GPUs where memory bandwidth is more constrained, the relative benefit of DES may differ substantially. Not tested on non-B200 hardware. NOT addressed.
2. **Block-size selection confound**: All Table 1 results use block length 32 (stated: "All experiments use a block length of 32 with 16 prefix and 16 suffix cache tokens"). The accuracy tradeoff at other block sizes is shown in Figure 7b but not the full Table 1 comparison vs baselines. NOT a major concern since Figure 7b provides robustness analysis.
3. **Single-GPU vs multi-GPU**: The paper's core motivation is multi-GPU/CPU-offloading scenarios. Single-GPU evaluation doesn't establish the multi-GPU benefit. NOT ruled out.

## Mechanism-attribution audit
- Two strategies (DES-Seq, DES-Vote) are both evaluated
- DES-Vote's advantage over DES-Seq is clearly shown (Table 1, Figure 4)
- Custom kernel overhead is separately profiled (Figure 6)
- No coupled-novelty attribution issue — the components are cleanly isolated

## Framing-vs-demonstration check
- Abstract: "latency by up to 38%"
- Body: "DES-Vote reduces **MoE layer latency** by up to 38.0%...Total **end-to-end GPU kernel time** improves by **8.2–14.3%**" (Sec 5.3)
- Gap: 38% (component) vs 8-14% (end-to-end) — 3-5x difference
- The paper acknowledges the dilution ("expected dilution stems from constant non-MoE operations like self-attention") but the abstract uses unqualified "latency"
- The multi-GPU motivation vs single-GPU evaluation: "multi-GPU and CPU-offloading environments" as the stated context; all measurements on a single B200

## Headline-number denominator audit
- "latency by up to 38%": denominator is MoE kernel latency, not end-to-end latency
- This is a component-level speedup presented as an unqualified "latency" claim in the abstract
